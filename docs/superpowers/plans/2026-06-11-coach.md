# Coach (Phase 5, engine-only) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** An opt-in, engine-only coach on web + iOS — NO LLM anywhere. Live coach (off by default): an eval bar beside the board updated asynchronously after each move, plus a dismissible banner after a player's mistake/blunder with a **template** explanation derived from concrete board checks (hung piece, missed capture, missed mate, walked into mate; generic material fallback), and a one-line "why" under the existing hint. Post-game review (from the result screen and from past-game replays): every position analyzed sequentially with a progress bar, producing per-move classification badges in the replay viewer, an accuracy %, and a top-3 "key moments" list. All analysis comes from a new `POST /v1/analyze` on the FastAPI engine (proxied by the Cloudflare Worker exactly like `/v1/move`), with the same fallback chain as the bot: web falls back to the local minimax `evaluate()`; iOS falls back remote → local Fairy-Stockfish → ChessBot minimax.

**Architecture:** Phase 5 EXTENDS the existing engine stack — no new storage, no new stacks. `server/main.py` gains `/v1/analyze` reusing the same `EngineManager`/`UCIEngine` Fairy-Stockfish subprocess (variant `chessborder`, same `variants.ini`, same client-centered FEN — **do not remap coordinates**). The worker (`server/worker/`) gains a forwarding route with the same `X-API-Key` injection, retry pattern, and movetime cap. CloudFront (`server/aws/static-site.yaml`) gains a `/v1/analyze` behavior (today only `/v1/move` and `/health` reach the worker — without this the route 404s in production) and the WAF rate rule widens from `/v1/move` to `/v1/`. Web gets a `src/coach/` package of pure modules (classify / explain / analyzeClient / review) wired into the existing `GameScreen`; post-game review is an extension of `renderReplay` (Phase 3's replay viewer). iOS mirrors the same modules in Swift with mirrored test fixtures.

## Settled decisions encoded in this plan (do not relitigate)

- **Engine-only.** No LLM, no new storage, no new infra stacks. Explanations are deterministic templates from board checks + engine eval/PV metadata.
- **Score perspective convention, fixed at each layer:**
  - **Server `/v1/analyze` returns side-to-move perspective** (UCI native — positive = good for the side to move). `mate_in` likewise: positive = side to move mates in N.
  - **Web/iOS classify/explain operate White-relative.** `analyzeClient` returns the server values untouched; `toWhiteRelative()` (in `classify.ts`, tested) converts at the boundary. Eval bar and review store White-relative `PositionEval`s.
- **Classification thresholds (centipawn swing, mover's perspective, after clamping non-mate evals to ±1500):** `ok < 50`, `50 ≤ inaccuracy < 150`, `150 ≤ mistake < 300`, `blunder ≥ 300`. Mate handling (exact rules in Task 1): losing a forced mate you had → `mistake` if still winning ≥ +300, else `blunder`; newly walking into a forced mate against you → `blunder` unless the position was already lost (< −300, then `ok`); mate-distance drift while still mating → falls through to the cp path (≈ 0 swing → `ok`).
- **Explanation check order (first hit wins):** walked-into-mate → missed-mate → hung piece → missed capture → generic fallback. Mate checks are **metadata-driven** (engine eval/PV — "mate in pv" per spec); hung/missed-capture are **board-driven** via the local rules engine (`fromFEN` + `legalMoves`), using legal-move attack counting (documented approximation: pinned attackers are ignored).
- **`/v1/analyze` movetime is capped server-side at 1000 ms** (Pydantic `le=1000`) and worker-side at `min(PUBLIC_MAX_MOVETIME_MS, 1000)`. Live coach uses 400 ms, post-game review 200 ms/position — a full 80-ply review ≈ 16 s of engine time, comfortably inside App Runner 0.5 vCPU autoscale 1–3 × MaxConcurrency 8. Review requests are **sequential** (never parallel) to stay polite to the single shared engine process.
- **Analyze requests accept `{fen?, moves?}` — clients only ever send `fen`.** `moves` (applied from the variant startpos, or from `fen` when both present) exists for parity with UCI `position` and future use; it forces a larger body cap (16 KB) than `/v1/move`'s 4 KB. The `/v1/move` client-parity check (`client_accepts_move`) is applied to the analyze best move only in the plain-`fen` case; on parity failure the response keeps the score but nulls `best_move_uci`/`pv` (analysis must not 422 a whole review).
- **Terminal positions are never analyzed.** Both clients guard on `result.type !== "ongoing"` per position (review analyzes positions 0..N−1 plus the final position only when the game ended by resignation — checkmate/stalemate final positions are skipped and their classification derived from the result). Defensively, `toWhiteRelative` maps the engine's `mate 0` (side to move is mated) to cp `∓MATE_CP` so nothing NaNs.
- **One in-flight analysis at a time, latest-wins** (live coach): token + `AbortController`, exactly the `requestHint` pattern already in `gameController.ts`. A stale response is dropped; the coach never blocks play, never throws into the game loop, and every failure degrades to "eval bar hides / no banner".
- **Coach toggle:** web `localStorage` key `chessborder.coachEnabled` (`"1"` = on; absent/anything else = off — off by default), surfaced in the existing settings panel under the board themes. iOS `@AppStorage("coachEnabled")` (default `false`) with a `Toggle` in `SettingsView`. The toggle takes effect immediately (next move), no reload.
- **Live coach scope: offline modes only** (`vsBot`, `localTwoPlayer`). Online games get **no** live coach (an engine whispering evals during a human game is cheating-adjacent) — flagged deliberate; post-game review of finished online games via Past Games replay is fine. In `vsBot`, the banner classifies only the player's (white's) moves; in `localTwoPlayer` it classifies both sides. The eval bar updates after every ply regardless of mover.
- **Banner is dismissible and transient:** `.coach-banner` under the status bar with a `×`; replaced/cleared automatically on the next move; only shown for `mistake`/`blunder` (inaccuracies tint the move-list badge in review but never interrupt live play).
- **Hint "why" is synchronous** from already-cached analysis: when the coach is enabled and a hint is shown, `hintWhy()` formats one line from the cached eval of the current position (mate → "Mates in N", capture → "Wins the ⟨piece⟩", check → "Forcing check", else "Engine's top move (+x.x)"). No extra network call; without a cached eval the line simply doesn't render.
- **Accuracy formula (exact, per side):** penalties `ok=0, inaccuracy=10, mistake=25, blunder=50`; `accuracy = max(0, 100 − round(Σpenalty / max(1, movesBySide)))`, displayed as a whole percent. **Key moments:** the top 3 plies by swing with classification ≠ `ok`, descending; each shows ply, move played, best move, and swing.
- **Post-game review entry points:** a "Review" button on the game-over overlay (live screen, works because stepping already works on finished games) and an "Analyze game" button in replay mode (`renderReplay` gains an options arg — backward-compatible). Review never auto-starts (it costs ~16 s of engine time); progress bar + cancel; results cached on the screen instance (re-entering replay re-analyzes — no persistence, flagged simplification).
- **Local fallback synthesizes analysis** when the server is unreachable: `scoreCp = evaluateCp(game, sideToMove)` (the existing minimax `evaluate()`, exported — its units are already ≈ centipawns: pawn = 100), `bestMoveUci` from the existing Web-Worker bot (`chooseLocalBotMove`, difficulty `hard`, randomness 0), `mateIn = null`, `pv = [best]`, `source: "local"`. iOS chain mirrors `HybridBotPlayer`'s platform order (iOS: remote → local FSF; Mac/simulator: local FSF → remote; then ChessBot minimax).
- **CloudFront + WAF are real deploy steps:** the new `/v1/analyze` CacheBehavior clones `/v1/move`'s, and the WAF rate rule's `SearchString` widens to `/v1/` (STARTS_WITH) so analyze traffic is rate-limited too. A review burst (~80 sequential requests over ~20 s) is far below the default `WafRateLimit` (per-IP per 5 min); call this out at deploy, don't tune the limit in this plan.
- **Engine fixture honesty:** hand-written Border Chess FENs in test fixtures must be validated in the tests themselves via `fromFEN` + `legalMoves` (the played and best moves must be legal). Mate metadata in fixtures is fixture input, not engine truth; before shipping, spot-check the two MISSED_MATE fixtures against the bundled Mac fairy-stockfish (see memory: FSF parity check, `go perft 1` / `go movetime`) and adjust the metadata if Border rules (border-square escapes!) diverge — border ranks/files mean classic back-rank patterns may not be mate.
- **Worker gets no test runner** (matching today's package); its `parseAnalyzePayload` mirrors `parseMovePayload` and reuses the same FEN regexes; correctness is covered by typecheck + the server's own validation + the verify-site smoke.
- **iOS FairyStockfishBot grows an `analyse` method** (it already speaks UCI); ChessBot's private `evaluate` becomes internal `evaluateCp` (mirroring the web export). New Swift coach files live in `ChessBorder/ChessBorder/Coach/`; xcodegen picks the folder up on regenerate. All coach unit tests must be Mac-runnable (no UIKit) so they run in the ChessBorderMac suite.

## API contract (single source of truth for all packages)

```
POST /v1/analyze        (worker → engine; worker injects X-API-Key; CORS like /v1/move)
  body: {
    fen?: string,         // client-centered chessborder FEN (validate_fen rules)
    moves?: string[],     // UCI from startpos (or from fen when both present), ≤1024,
                          // each matching ^[a-j](10|[1-9])[a-j](10|[1-9])[qrbn]?$
    movetime_ms?: int     // default 200; server clamps 50..1000; worker clamps
  }                       //   min(PUBLIC_MAX_MOVETIME_MS, 1000)
  → 200 {
    score_cp:      int|null,    // side-to-move perspective; null when mate_in set
    mate_in:       int|null,    // side-to-move: +N = mover mates in N
    best_move_uci: string|null, // null on terminal positions or parity rejection
    pv:            string[]     // [] when best_move_uci is null
  }
  | 400 invalid fen/moves | 401 bad key (direct) | 413 body > 16 KB | 503 engine down
```

Requires `fen` or `moves` (400 otherwise). Clients in this plan always send `fen` only.

```ts
import type { PieceColor } from "../engine/types";

export type MoveClassification = "ok" | "inaccuracy" | "mistake" | "blunder";

/** White-relative engine evaluation. Exactly one field is non-null. */
export interface PositionEval {
  /** Centipawns, White-relative. Null when the position is a forced mate. */
  cp: number | null;
  /** Mate distance, White-relative: +N = White mates in N, -N = Black mates in N. */
  mateIn: number | null;
}

export const MATE_CP = 10_000;
export const CLAMP_CP = 1_500;
/** A mover still at/above this (own-perspective cp) after missing a mate is a mistake, not a blunder. */
export const WINNING_CP = 300;
export const INACCURACY_CP = 50;
export const MISTAKE_CP = 150;
export const BLUNDER_CP = 300;

/** Convert a server analysis (side-to-move perspective) to White-relative.
 *  mate_in === 0 (side to move is already mated) maps defensively to ∓MATE_CP cp —
 *  callers should never analyze terminal positions, but nothing may NaN if they do. */
export function toWhiteRelative(
  scoreCp: number | null,
  mateIn: number | null,
  sideToMove: PieceColor
): PositionEval {
  const sign = sideToMove === "white" ? 1 : -1;
  if (mateIn != null) {
    if (mateIn === 0) return { cp: sign * -MATE_CP, mateIn: null };
    return { cp: null, mateIn: mateIn * sign };
  }
  return { cp: (scoreCp ?? 0) * sign, mateIn: null };
}

/** Mate-aware, clamped White-relative cp used for swing arithmetic. */
function normalizedCp(e: PositionEval): number {
  if (e.mateIn != null) {
    const sign = e.mateIn > 0 ? 1 : -1;
    return sign * (MATE_CP - Math.abs(e.mateIn));
  }
  return Math.max(-CLAMP_CP, Math.min(CLAMP_CP, e.cp ?? 0));
}

/** Classify the move a player just made, from the evals before and after it.
 *  before/after are White-relative; mover is the side that made the move. */
export function classifyMove(
  before: PositionEval,
  after: PositionEval,
  mover: PieceColor
): MoveClassification {
  const sign = mover === "white" ? 1 : -1;
  const moverBefore = normalizedCp(before) * sign;
  const moverAfter = normalizedCp(after) * sign;

  const hadMate = before.mateIn != null && before.mateIn * sign > 0;
  const hasMate = after.mateIn != null && after.mateIn * sign > 0;
  const facedMate = before.mateIn != null && before.mateIn * sign < 0;
  const facesMate = after.mateIn != null && after.mateIn * sign < 0;

  if (hadMate && !hasMate) {
    return moverAfter >= WINNING_CP ? "mistake" : "blunder"; // missed a forced mate
  }
  if (facesMate && !facedMate) {
    return moverBefore >= -WINNING_CP ? "blunder" : "ok"; // walked into mate (unless already lost)
  }

  const swing = Math.max(0, moverBefore - moverAfter);
  if (swing >= BLUNDER_CP) return "blunder";
  if (swing >= MISTAKE_CP) return "mistake";
  if (swing >= INACCURACY_CP) return "inaccuracy";
  return "ok";
```

```python
def parse_analysis(lines: list[str]) -> dict:
    """Parse UCI `info ... score ... pv ...` output into an analysis dict.

    Scores are from the side-to-move's perspective (UCI native). The last
    exact (non-bound) info line wins; `bestmove (none)` means a terminal
    position (no score, no pv).
    """
    score_cp: int | None = None
    mate_in: int | None = None
    pv: list[str] = []
    best: str | None = None
    for line in lines:
        parts = line.split()
        if not parts:
            continue
        if parts[0] == "bestmove":
            best = parts[1] if len(parts) > 1 and parts[1] != "(none)" else None
            continue
        if parts[0] != "info" or "score" not in parts or "pv" not in parts:
            continue
        if "lowerbound" in parts or "upperbound" in parts:
            continue
        si = parts.index("score")
        kind, value = parts[si + 1], int(parts[si + 2])
        if kind == "cp":
            score_cp, mate_in = value, None
        elif kind == "mate":
            score_cp, mate_in = None, value
        else:
            continue
        pv = parts[parts.index("pv") + 1 :]
    if best is None:
        pv = []
    return {"score_cp": score_cp, "mate_in": mate_in, "best_move_uci": best, "pv": pv}
```

```ts
export const ANALYZE_MAX_MOVETIME_MS = 1000;
const ANALYZE_MAX_BODY = 16_384;
const MAX_ANALYZE_MOVES = 1024;
const UCI_MOVE = /^[a-j](?:10|[1-9])[a-j](?:10|[1-9])[qrbn]?$/;

export type AnalyzePayload = {
  fen?: string;
  moves?: string[];
  movetime_ms: number;
};

export function parseAnalyzePayload(raw: string): AnalyzePayload | { error: string } {
  if (raw.length > ANALYZE_MAX_BODY) return { error: "Request body too large" };

  let body: unknown;
  try {
    body = JSON.parse(raw);
  } catch {
    return { error: "Invalid JSON" };
  }
  if (!body || typeof body !== "object") return { error: "Invalid JSON body" };
  const record = body as Record<string, unknown>;

  let fen: string | undefined;
  if (record.fen != null) {
    if (typeof record.fen !== "string") return { error: "Invalid FEN" };
    const trimmed = record.fen.trim();
    if (!trimmed || trimmed.length > MAX_FEN_LENGTH) return { error: "Invalid FEN" };
    if (/[\n\r ]/.test(trimmed)) return { error: "Invalid FEN characters" };
    const normalized = trimmed.replace(/\s+/g, " ");
    if (!FEN_BOARD.test(normalized) || !FEN_TAIL.test(normalized)) {
      return { error: "Invalid FEN format" };
    }
    fen = normalized;
  }

  let moves: string[] | undefined;
  if (record.moves != null) {
    if (!Array.isArray(record.moves) || record.moves.length > MAX_ANALYZE_MOVES) {
      return { error: "Invalid moves" };
    }
    if (!record.moves.every((m) => typeof m === "string" && UCI_MOVE.test(m))) {
      return { error: "Invalid moves" };
    }
    moves = record.moves as string[];
  }

  if (!fen && !moves) return { error: "fen or moves is required" };

  const movetimeMs = record.movetime_ms ?? 200;
  if (
    typeof movetimeMs !== "number" ||
    !Number.isFinite(movetimeMs) ||
    movetimeMs < 50 ||
    movetimeMs > 30_000
  ) {
    return { error: "Invalid movetime_ms" };
  }

  return { fen, moves, movetime_ms: movetimeMs };
}
```
